Changes since 2017_12_24

2026_03_15 -- TDLeaf: separate PSQT Adam LR + FC int8 float-shadow clamping (nnue.cpp, tdleaf.h).
               Two fixes diagnosed from 5,000-game training analysis:
               (1) FC0/FC1 float-shadow clamping: after each weight update, the float shadow
               is clamped to [-127, +127] (the int8 quantisation boundary).  Without this,
               the float shadow can drift arbitrarily beyond ±127 while the requantised
               inference weight is stuck at the boundary — a "zombie" weight that accumulates
               gradient with no effect on the network.  This caused the observed FC1 pileup
               at the int8 extremes (std 49.8, range ±155 vs int8 limit ±127).  Not applied
               to FC2 (output layer, no activation clamping).
               (2) Separate PSQT Adam LR0 (TDLEAF_ADAM_PSQT_LR0 = 20.0): Adam normalises
               gradient magnitude, so only LR0 controls the per-step size in weight-space.
               PSQT weights are at int32 scale (std ~36,000) while FC weights are at int8
               scale (std ~3-50) — a ratio of ~1,000×.  Using the same LR0=0.5 for both
               caused PSQT to learn ~1,000× slower than FC per update.  NNUE_PSQT_LR_SCALE
               is ineffective with Adam (normalised away) and only the LR0 matters.
               PSQT_LR0=20 gives ~5% baseline change per feature over 5,000 games vs ~0.3%
               previously.

2026_03_15 -- TDLeaf Adam + per-weight LR decay (nnue.cpp, tdleaf.h).
               nnue_apply_gradients() now supports Adam with per-weight learning-rate
               decay as an alternative to plain gradient descent.  Controlled by
               TDLEAF_ADAM_LR0 in tdleaf.h (set to 0.0 for plain GD; default 0.5).
               Algorithm:
                 lr(cnt) = LR0 / (1 + cnt / C)     (C = TDLEAF_ADAM_C = 500)
               Fast initial learning when cnt ≈ 0; decays to near-zero as the weight
               converges.  Uses existing per-weight cnt arrays (already persisted in
               .tdleaf.bin), so no sidecar file is needed.  Session-local moments (v, m)
               and step counter t_adam are reset to zero at each session start alongside
               the v/m arrays, keeping bias correction v/(1-β²^t) always valid.
               Layer breakdown:
                 FC layers (FC0/FC1/FC2) and FT biases: full Adam (per-weight m and v).
                 FT weights: RMSProp with per-row v (92 MB per-weight v ruled out;
                   per-row v ≈ 88 KB).  No per-row m (directionally meaningless as scalar).
                 PSQT: full Adam per-weight (only 8 buckets/row; per-weight v ≈ 1.4 MB).
               Hyperparameters added to tdleaf.h: TDLEAF_ADAM_LR0, TDLEAF_ADAM_C,
               TDLEAF_ADAM_BETA1, TDLEAF_ADAM_BETA2, TDLEAF_ADAM_EPS.

2026_03_15 -- nnue_init_zero_weights: He-inspired weight initialisation (nnue.cpp).
               Two changes to the fresh-network init used by --init-nnue:
               (1) All FC weight means zeroed (were inherited from Stockfish 15.1
                   endpoint: FC0 +0.24, FC1 -1.10, FC2 +1.10, FT -0.71).  Non-zero
                   means create a systematic directional bias TDLeaf must undo before
                   learning signal; zero mean (the He/Kaiming principle) is the correct
                   starting point.
               (2) FC0 std reduced 8.4252 → 3.0.  FC0 has 1024 inputs; with the actual
                   SqrCReLU input statistics (E[x²] ≈ 2200, inputs in [0,127]) the
                   old σ=8.4 gave std(raw>>6) ≈ 197 and ~24% neuron saturation.
                   σ=3.0 gives std(raw>>6) ≈ 70 and ~3% saturation, matching the He
                   design target.  FC1 (30 inputs) and FC2 (output layer) stds unchanged.
               FT std 44.4 and all bias zero-init unchanged.

2026_03_15 -- nnue_init_zero_weights: PSQT initialised from classical piece-square
               tables (nnue.cpp).  Each of the 8 PSQT buckets is now seeded with
               material + piece-square positional bonus from score.h piece_sq[4][7][64],
               interpolated to the appropriate classical game stage:
               Bucket 0 → pure opening (stage 0); buckets 1-5 → interpolated transitions;
               buckets 6-7 → endgame (stage 3).  White pieces use whitef[] rank-flip;
               black pieces index directly.  Scale: cp × 5776/100 (NNUE int32 units).
               Previously all 8 buckets were identical (uniform material, no PSQ).

2026_03_14 -- TDLeaf: non-blocking file lock in nnue_save_fc_weights() (nnue.cpp).
               Previously used LOCK_EX (blocking), which serialised all 10 concurrent
               training processes at game end — causing the next game to start while the
               engine was still blocked, charging the delay to white's clock and producing
               ~36–40 time forfeits per 100 games at 0:03+0.05.
               Now uses LOCK_EX | LOCK_NB: if the lock is held by another process, the
               save is skipped and a "deferring save" message is logged to stderr.
               Accumulated weight deltas are retained in memory and flushed on the next
               successful save, so no training signal is lost.
               Result: 0 time forfeits in 100-game test (down from ~40).

2026_03_14 -- Time management: raise MOVE_OVERHEAD_CS 2 → 5 (search.h, search.cpp).
               50 ms is more representative of real-world interface/OS latency.
               min_cushion floor in search.cpp is now derived from MOVE_OVERHEAD_CS
               directly (was hardcoded 3) so both mechanisms stay in sync.

2026_03_14 -- Time management: buffer for small-increment time controls (search.h, search.cpp, main.cpp).
               Prevents time forfeits at fast controls with increment < 0.1 seconds.
               Three changes:
               (1) search.h: added MOVE_OVERHEAD_CS = 2 (centiseconds).  Tunable
                   constant for per-move interface/OS overhead, analogous to EXTEND_TIME_SCORE.
               (2) search.cpp: min_cushion floored at 3 cs (was 0 for base=0 games);
                   max_margin floored at 3*min_cushion so the short-time-control
                   activation threshold is non-zero for increment-only time controls.
               (3) main.cpp: when inc > 0 and inc < 0.10 s, MOVE_OVERHEAD_CS is
                   subtracted from time_limit before each search (clamped to MIN 1 cs).
                   For 0+0.05 this reduces a typical 5 cs budget to 3 cs, leaving
                   ~20 ms headroom for the interface round-trip per move.

2026_03_11 -- Flavor B epoch-based replay for TDLeaf(λ) (TDLEAF=1).
               After each game, the completed TDGameRecord is pushed into a
               static ring buffer (TDLEAF_REPLAY_BUF_N=8 games, ~36 MB BSS).
               tdleaf_replay() then runs TDLEAF_REPLAY_K additional passes over
               all buffered games: score_stm is refreshed from the current
               quantized weights (nnue_evaluate_acc_raw) before each pass, then
               gradients are accumulated across all games and applied once per pass.
               New functions: nnue_evaluate_acc_raw (nnue.cpp/h),
               tdleaf_accumulate_game, tdleaf_refresh_scores, tdleaf_replay (tdleaf.cpp/h).
               tdleaf_update_after_game refactored to call tdleaf_accumulate_game.
               Build flags: TDLEAF_REPLAY_K (default 2), TDLEAF_REPLAY_BUF_N (default 8).
               Ablation results: K=2 best; K=3 marginally worse; K=0 much worse; K=6 large regression.

2026_03_11 -- TDLeaf horizon noise mitigation: two mechanisms to reduce the influence
               of tactics beyond the search horizon on gradient updates.
               Approach 1 — TD error clipping (TDLEAF_SCORE_CLIP_CP):
               When the white-POV score change between consecutive moves exceeds the
               threshold, the (d[t+1]-d[t]) eligibility trace contribution is scaled
               down proportionally.  Direction preserved; only magnitude reduced.
               Approach 2 — ID score stability weighting (TDLEAF_ID_VAR_SIGMA2):
               The last TD_ID_HIST=4 ID scores are tracked per search; their variance
               is stored in TDRecord::id_score_variance and used to soft-weight each
               ply's gradient by 1/(1 + variance/TDLEAF_ID_VAR_SIGMA2).
               Both approaches apply in replay passes as well as the live pass.
